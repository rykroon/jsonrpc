Batch support (deferred)

Lives on Server.ServeMessage. Sketch: detect '[' as the first non-space
byte, decode as []Request, call Serve per element, collect non-nil
Responses, marshal as a JSON array. An all-notifications batch returns
(nil, nil) per spec.

Open questions to resolve when implementing:
  - Batch-level observability (logging size on receipt). Add a separate
    method or let users wrap.
  - Whether to enforce a max batch size.


Parse errors and layering

JSON-RPC's CodeParseError isn't universally right — it's a layering
mistake for transports with native error signaling (HTTP -> 400) and the
correct answer for transports that don't (stdio, TCP stream, single
WebSocket frame in a long-lived connection). Don't centralize
parse-error response generation at the Server level. Each transport
adapter picks its idiom: HTTP returns 400, stream transports use
Server.ServeMessage's in-band CodeParseError.


Sender / Transport asymmetry

Sender (client side) = the transport. Same shape on both sides of any
real wire. In-process is the degenerate case: a direct function call,
exposed via InProcess(s).

Server side has no symmetric single-interface "ServerTransport" because
transport shapes diverge: HTTP is a one-shot handler, stdio/TCP/WebSocket
is a long-lived loop, in-process is nothing. net/rpc's codec picked the
streaming shape and that's why net/rpc/jsonrpc is incomplete (no real
notifications, no batch, no error.data).

If a common server-transport shape ever emerges from writing real
adapters, name it then.
